Inference to broader populations

Why do you restrict it to just the spine? That means some data goes missing.

We wanted this tool to help explore specifically the data that is linked across datasets, to get a complete picture of those people for whom we have reasonably reliable data. The only safe way to do this and avoid double-counting people is to drop people who have not been linked to the spine. Yes, this means that for any specific dataset we have filtered out a range of data. We have crude estimates of the linkage rates for each variable under the "variable"" tab; these represent the minimum linkage rate of the tables that were used in creating the particular variable. This needs to be carefully taken into account in any inference - there are missing linkages that have not been corrected for.

If you want to analyse a particular dataset as fully as possible eg ACC Claims so you can see all ACC Claims not just those linked to the spine, you need to go back to the original IDI data, which of course has it all there.

Why don't you restrict it to just the estimated resident population in any year?

We have made the estimated resident population ("IDI-ERP") the default, but have included data from others. This lets us go back further than the current estimated resident population table, and also lets people consider alternative versions (for, example filtering by a particular number of days spent in New Zealand and activity from a particular admin data source).

Why are some of the linkage rates in the 'variables' page so low? (eg about 15% for many)

Linkage rates have been estimated by taking the minimum of the linkage rate for the source tables directly used in the creation of a variable. In the case of variables like those relating to ethnicity, birth year, sex etc, they have been drawn from IDI_Clean.data.personal_detail. Because that table is derived from many other tables, it has more than 60 million rows of potential people, and only around 9 million of them are joined to the spine. Hence the low reported linkage rate. Basically, the linkage rate isn't particularly meaningful in this situation, which is why in the explanatory text the tool produces, such variables are described as "enduring characteristics of people on the spine that are estimated with reasonable accuracy", rather than citing the actual linkage rate.

Linkage rates are most meaningful for data that has what we refer to as a "person-period" grain, meaning it is observations that change over time for individuals (such as ACC claims, income, and number of hospital discharges).

Tools, technology and definitions

Where can I see the source code?

The source code to build the database (`build-db`) and for this front end web application (`explorer-shiny`) is available on GitHub. Note that typically this is some weeks behind the latest version being developed in Stats NZ.

The code to define particular variables (eg income, mental health, etc) is in the `./build-db/fact-years/` sub-directory.
